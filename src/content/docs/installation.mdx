SnapDog is distributed as a single binary. It runs on Linux and macOS, and can be installed via package managers, run in Docker, or compiled from source.

---

## 1. Installation

Choose your preferred platform below to install SnapDog:

<Tabs>
  <TabItem label="Docker">
    Run the production container by binding your audio config file and mapping a data volume:
    
    ```yaml
    # docker-compose.yml
    services:
      snapdog:
        image: ghcr.io/metaneutrons/snapdog:latest
        restart: unless-stopped
        volumes:
          - snapdog-data:/var/lib/snapdog
          - ./snapdog.toml:/etc/snapdog/snapdog.toml:ro
        ports:
          - "5555:5555"      # WebUI + REST API
          - "1704:1704"      # Snapcast streaming
          - "3671:3671/udp"  # KNX/IP device
    
    volumes:
      snapdog-data:
    ```
  </TabItem>
  <TabItem label="Arch Linux (AUR)">
    Install the package directly from the Arch User Repository using your favorite AUR helper:
    
    ```bash
    yay -S snapdog snapdog-client
    ```
  </TabItem>
  <TabItem label="macOS (Homebrew)">
    Tap the repository and install the command-line utility:
    
    ```bash
    brew tap metaneutrons/tap
    brew install snapdog snapdog-client
    ```
  </TabItem>
  <TabItem label="Debian/Ubuntu">
    Add the stable APT repository:
    
    ```bash
    echo "deb [trusted=yes] https://metaneutrons.github.io/snapdog/debian stable main" \
      | sudo tee /etc/apt/sources.list.d/snapdog.list
    sudo apt update
    sudo apt install snapdog snapdog-client
    ```
  </TabItem>
  <TabItem label="Windows Service">
    SnapDog can run as a native Windows background service:
    
    ```cmd
    sc create SnapDog binPath= "\"C:\Program Files\SnapDog\snapdog.exe\" --service --config \"C:\ProgramData\snapdog\snapdog.toml\""
    sc start SnapDog
    ```
  </TabItem>
</Tabs>

## 2. Configuration (`snapdog.toml`)

SnapDog uses a single TOML configuration file. It follows **convention over configuration**—most naming structures, streaming ports, and automation channels are derived automatically from your zone and client definitions to keep the config file as small as possible.

Here is a complete, documented configuration template:

```toml
# Server display name — used for mDNS, MQTT, and browser titles
name = "SnapDog"

[http]
port = 5555
bind = "0.0.0.0"                       # Set :: for dual-stack IPv4+IPv6
base_url = "http://192.168.1.10:5555"  # Required for generating cover art URLs
# api_keys = ["secret1", "secret2"]   # Optional token protection

[audio]
sample_rate = 48000
bit_depth = 16                         # 16 or 24 (FLAC)
source_conflict = "last_wins"          # last_wins | receiver_wins
zone_switch_fade_ms = 300              # Fades when changing client zones (max 1000)
source_switch_fade_ms = 300            # Fades when changing active audio source

[snapcast]
streaming_port = 1704
unknown_clients = "accept"             # accept | ignore | reject
default_zone = "Living Room"           # Fallback zone for new clients

[dbus]
enabled = true                         # Enables MPRIS2 interface per zone (Linux-only)

[mqtt]
broker = "192.168.1.10:1883"
base_topic = "snapdog/"                # Namespace for commands & states

[subsonic]
url = "https://music.example.com"
username = "user"
password = "pass"
format = "flac"                        # flac | mp3 | opus
[subsonic.cache]
path = "~/.cache/snapdog/tracks"
max_size_mb = 2048                     # LRU cache eviction size

# Define Zones (physical rooms)
[[zone]]
name = "Living Room"
icon = "sofa"                          # Icon used in WebUI

[[zone]]
name = "Kitchen"
icon = "cook"

# Define Clients (speakers connected to Zones by MAC address)
[[client]]
name = "Living Room L"
mac = "00:11:22:33:44:55"
zone = "Living Room"

[[client]]
name = "Kitchen Speaker"
mac = "02:42:ac:11:00:10"
zone = "Kitchen"

# Define Internet Radio Stations
[[radio]]
name = "Deutschlandfunk"
url = "https://st01.sslstream.dlf.de/dlf/01/high/aac/stream.aac"
```

## 3. Server Backend Recommendations

SnapDog supports two Snapcast modes via compile-time features:

1. **`snapcast-embedded` (Default & Recommended):** Runs the Snapcast server directly inside the SnapDog process. It is the preferred method because:
   * It requires no external dependencies or processes to manage.
   * It fully supports custom protocol channels (message IDs 10–15) used to communicate parametric EQ adjustments and speaker-correction profiles to custom hardware.
   * It handles client volume and connection events inline, reducing metadata syncing delays.
2. **`snapcast-process`:** Spawns and manages an external C++ `snapserver` process. This is only recommended if you need to use specific C++ server features not yet ported to `snapcast-rs`.
