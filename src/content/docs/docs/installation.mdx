**SnapDog** is distributed as a lightweight, single statically-linked binary. It can be deployed on Linux (x86_64, aarch64, armv7) and macOS. Choose the installation method that fits your environment below.

---

## 1. System Requirements & Prerequisites

Before setting up SnapDog, ensure your target machine meets the following requirements:

### Operating System
* **Linux:** Kernel 5.4 or higher. ALSA (Advanced Linux Sound Architecture) is required for local audio output interfaces.
* **macOS:** macOS 12 (Monterey) or higher.
* **Windows:** Windows 10/11 or Windows Server 2019+ (supported when running as a background service).

### Network Ports
SnapDog binds to several ports depending on your configuration. Ensure your firewall permits traffic on these ports:
* `5555/tcp` — REST API, WebUI, and WebSockets (configurable via `[http].port`).
* `1704/tcp` — Snapcast streaming endpoint where clients connect.
* `1705/tcp` — Snapcast JSON-RPC control API (only for external snapcast).
* `3671/udp` — KNX/IP routing or tunneling communication (if KNX is enabled).
* `5353/udp` — mDNS/Bonjour discovery advertisement.

## 2. Installation Methods

<Tabs>
  <TabItem label="Docker (Compose)">
    Docker is the recommended deployment method for home servers or NAS devices (like Unraid or Synology).

    1. Create a `docker-compose.yml` file:
       ```yaml
       version: '3.8'
       services:
         snapdog:
           image: ghcr.io/metaneutrons/snapdog:latest
           container_name: snapdog
           restart: unless-stopped
           network_mode: host # Required for mDNS discovery and KNX multicast
           devices:
             - "/dev/snd:/dev/snd" # Expose ALSA soundcards for direct playback
           volumes:
             - snapdog-data:/var/lib/snapdog
             - ./snapdog.toml:/etc/snapdog/snapdog.toml:ro
           environment:
             - TZ=Europe/Berlin

       volumes:
         snapdog-data:
       ```
    2. Start the container:
       ```bash
       docker compose up -d
       ```
  </TabItem>

  <TabItem label="Arch Linux (AUR)">
    AUR helper tools can be used to install the latest pre-compiled binary or build SnapDog directly from the Arch User Repository.

    ```bash
    # Install the server and control client
    yay -S snapdog snapdog-client
    
    # Enable and start the systemd service
    sudo systemctl enable --now snapdog
    ```
  </TabItem>

  <TabItem label="macOS (Homebrew)">
    For Apple Silicon or Intel Macs, we maintain a custom Homebrew tap:

    ```bash
    # Tap the repository
    brew tap metaneutrons/tap

    # Install the server and control utility
    brew install snapdog snapdog-client

    # Run SnapDog as a macOS background service
    brew services start snapdog
    ```
  </TabItem>

  <TabItem label="Debian/Ubuntu (APT)">
    We distribute native `.deb` packages for Debian and Ubuntu architectures (amd64, arm64).

    1. Register the trusted APT repository source:
       ```bash
       echo "deb [trusted=yes] https://metaneutrons.github.io/snapdog/debian stable main" \
         | sudo tee /etc/apt/sources.list.d/snapdog.list
       ```
    2. Update package lists and install:
       ```bash
       sudo apt update
       sudo apt install snapdog snapdog-client
       ```
    3. Start the background service:
       ```bash
       sudo systemctl daemon-reload
       sudo systemctl enable --now snapdog
       ```
  </TabItem>

  <TabItem label="Windows Service">
    SnapDog compiles to a native Windows binary and can run as an NT Service.

    1. Download the Windows binary (`snapdog.exe`) and place it in `C:\Program Files\SnapDog\`.
    2. Open a Command Prompt as Administrator and register the service:
       ```cmd
       sc create SnapDog binPath= "\"C:\Program Files\SnapDog\snapdog.exe\" --service --config \"C:\ProgramData\snapdog\snapdog.toml\"" start= auto
       sc start SnapDog
       ```
  </TabItem>
</Tabs>

## 3. Direct Binary Download

If you prefer to install the pre-compiled binaries manually:

1. Download the correct archive for your architecture from the [GitHub Releases](https://github.com/metaneutrons/snapdog/releases) page.
2. Extract the archive and copy the binary to `/usr/local/bin`:
   ```bash
   tar -xzf snapdog-linux-amd64.tar.gz
   sudo mv snapdog /usr/local/bin/
   sudo chmod 755 /usr/local/bin/snapdog
   ```

## 4. Compiling from Source

If you want to compile SnapDog with custom target optimizations, you can build it from source using Cargo.

<Aside type="note">
  Building from source allows choosing specific Snapcast server backends. See the compilation feature flags described in the [Server Mode Selection section](#6-server-mode-selection) below.
</Aside>

### Build Dependencies
Ensure your development machine has the required system libraries installed:
* **Debian/Ubuntu:** `sudo apt install build-essential pkg-config clang libasound2-dev libavahi-compat-libdnssd-dev`
* **macOS:** Xcode Command Line Tools are sufficient.

### Building
1. Clone the repository:
   ```bash
   git clone https://github.com/metaneutrons/snapdog.git
   cd snapdog
   ```
2. Build the release binary:
   ```bash
   cargo build --release --bin snapdog
   ```
3. The compiled binary will be available at `target/release/snapdog`.

## 5. Post-Installation Service Scripts

To ensure SnapDog starts automatically on boot and restarts in case of failures, configure a service supervisor.

### Linux Systemd Service Unit
Create `/etc/systemd/system/snapdog.service`:

```ini
[Unit]
Description=SnapDog Multi-room Audio Controller
After=network-online.target sound.target
Wants=network-online.target

[Service]
Type=simple
User=snapdog
Group=audio
ExecStart=/usr/local/bin/snapdog --config /etc/snapdog/snapdog.toml
Restart=always
RestartSec=5
LimitRTPRIO=99
LimitMEMLOCK=infinity

[Install]
WantedBy=multi-user.target
```

### macOS Launchd Service Agent
Create `~/Library/LaunchAgents/cc.snapdog.server.plist`:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>Label</key>
    <string>cc.snapdog.server</string>
    <key>ProgramArguments</key>
    <array>
        <string>/usr/local/bin/snapdog</string>
        <string>--config</string>
        <string>/Users/shared/snapdog.toml</string>
    </array>
    <key>RunAtLoad</key>
    <true/>
    <key>KeepAlive</key>
    <true/>
</dict>
</plist>
```

## 6. Server Mode Selection

SnapDog supports two primary Snapcast compilation backends. You can select the backend using Cargo feature flags:

* **`snapcast-embedded` (Default & Recommended):** Integrates the Snapcast server directly into the Rust daemon binary (leveraging our native Rust port of the original C++ Snapcast codebase via the `snapcast-*` crates). This mode is strongly recommended because it manages group synchronization inline, supports proprietary float PCM channels for parametric EQ corrections, and doesn't require maintaining an external binary.
* **`snapcast-process`:** Spawns and manages a standalone C++ `snapserver` process in the background. Choose this only if you must rely on advanced C++ features or legacy external pipe encoders.
