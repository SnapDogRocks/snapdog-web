This guide documents the command-line interfaces for both the **SnapDog Server (`snapdog`)** and the **SnapDog Client (`snapdog-client`)**.

---

## 1. SnapDog Server

The `snapdog` server runs as a daemon that manages audio streams, routes signals to zones, and acts as the central coordinator.

<Aside type="note">
  For running the server as a background service supervised by systemd or launchd, see the [Installation Guide](/docs/installation#5-post-installation-service-scripts).
</Aside>

### Required Execution Modes

At startup, the server requires you to declare its primary configuration mode. You **must** specify either `--config` or `--knx-device`.

* **Configuration File Mode:** Standard operation using a TOML file.
  ```bash
  snapdog --config /path/to/snapdog.toml
  ```
* **KNX Device Mode:** Bootstraps with standard defaults and relies on an ETS (Engineering Tool Software) project download over KNX routing/tunneling to configure zones and group addresses.
  ```bash
  snapdog --knx-device
  ```

<Aside type="danger">
If neither `--config` nor `--knx-device` is specified, the server prints the help guide and exits immediately with status code `1`.
</Aside>

### Server Boot Examples

<Tabs>
  <TabItem label="Standard Daemon Mode">
    Launch the server on Linux or macOS with a specific configuration file:
    ```bash
    snapdog --config /etc/snapdog/snapdog.toml
    ```
  </TabItem>

  <TabItem label="Debugging & Port Override">
    Temporarily override the web server port and increase logging verbosity to debug connection issues:
    ```bash
    snapdog --config ~/.config/snapdog.toml --port 8080 --log-level debug
    ```
  </TabItem>

  <TabItem label="KNX Integration Bootstrapping">
    Launch in device mode to let an integrator assign a physical address (e.g., `1.1.25`) and program the server using the `.knxprod` file:
    ```bash
    snapdog --knx-device --knx-address 1.1.25 --knx-prog-mode
    ```
  </TabItem>
</Tabs>

## 2. SnapDog Client

The `snapdog-client` player runs on speaker endpoints, connecting back to a SnapDog server to pull and play synchronized PCM audio.

### Server Connection URL

The client takes an optional positional argument specifying the server connection string:
```bash
snapdog-client [URL]
```
The `URL` has the format `<scheme>://<host>[:port]`, where scheme can be `tcp`, `ws` (WebSockets), or `wss` (Secure WebSockets).
* **mDNS Discovery (Default):** If the URL argument is omitted, the client searches for a local server automatically using `tcp://_snapdog._tcp`.
* **Direct TCP Socket:** Connects directly to the stream socket (e.g. `tcp://192.168.1.50:1704`).
* **WebSocket Stream:** Connects over HTTP WebSockets (e.g. `ws://192.168.1.50:1780` or secure `wss://myname.cc`).

### Client Arguments Reference

| Option | Shorthand | Type | Description |
| :--- | :--- | :--- | :--- |
| `[URL]` | â€” | Positional | Connection URL. Defaults to `tcp://_snapdog._tcp` (mDNS). |
| `--instance <ID>` | `-i` | Integer | Instance ID when running multiple client players on the same host (default: `1`). |
| `--hostID <ID>` | â€” | String | Unique host identifier. Defaults to the network card's MAC address. |
| `--soundcard <NAME>` | `-s` | String | PCM audio output device name or index. Defaults to `default`. |
| `--list` | `-l` | Flag | Lists all available local PCM soundcard output devices and exits. |
| `--test-tone` | â€” | Flag | Plays a brief audio test tone and exits to verify soundcard output. |
| `--latency <MS>` | â€” | Integer | Manual latency offset in milliseconds to align with laggy hardware (default: `0`). |
| `--sampleformat <FMT>` | â€” | String | Force resampler parameters in format `<rate>:<bits>:<channels>` (e.g., `48000:16:*`). |
| `--player <BACKEND>` | â€” | String | Audio player backend and optional arguments in format `<name>[:<params>\|?]` (e.g. `alsa`). |
| `--mixer <MODE>` | â€” | String | Volume mixer mode. Options: `software` (default), `hardware:<ctrl>`, `midi:<params>`, or `none`. |
| `--cert <PATH>` | â€” | String (Path) | Client certificate file (PEM format) for secure connections. |
| `--cert-key <PATH>` | â€” | String (Path) | Client private key file (PEM format) for secure connections. |
| `--key-password <PW>` | â€” | String | Password for encrypted private keys. |
| `--server-cert <PATH>`| â€” | String (Path) | CA certificate file to verify the server certificate. |
| `--encryption-psk <KEY>`| â€” | String | Pre-shared key for stream decryption when using `f32lz4e`. |
| `--daemon [PRIORITY]` | `-d` | Integer | Run in the background (daemonized) with optional process priority `[-20..19]`. *(Unix only)* |
| `--user <USER[:GROUP]>`| â€” | String | System user and group to run as when daemonized. *(Unix only)* |
| `--logsink <SINK>` | â€” | String | Log sink output target: `null`, `system`, `stdout`, `stderr`, or `file:<path>`. |
| `--logfilter <FILTER>`| â€” | String | Log tag levels filter (default: `*:info`, e.g. `*:debug`). |

<Aside type="tip">
**Mixer Modes:**
* `software` (Default): Digital volume scaling of the incoming PCM buffer.
* `hardware:<control>`: Calls ALSA's mixer interface to set physical soundcard gain (e.g. `--mixer hardware:Master` or `--mixer hardware:Speaker`).
* `midi:<interface>:<ch>`: Maps incoming volume levels to a MIDI CC control message (typically CC7).
* `none`: Disables volume adjustments on the client.
</Aside>

### Client Boot Examples

<Tabs>
  <TabItem label="Standard Auto-Discovery">
    Start playing. Searches network for a SnapDog server automatically:
    ```bash
    snapdog-client
    ```
  </TabItem>

  <TabItem label="Hardware Mixer Control">
    Directly target a server IP and utilize ALSA physical card mixers to control volume:
    ```bash
    snapdog-client --soundcard "hw:CARD=DAC,DEV=0" --mixer "hardware:Digital" tcp://192.168.1.100:1704
    ```
  </TabItem>

  <TabItem label="Diagnostics Check">
    List all local audio output interfaces, run a test sound, and exit:
    ```bash
    snapdog-client --list
    snapdog-client --soundcard "hw:1" --test-tone
    ```
  </TabItem>

  <TabItem label="Secure WebSockets & PSK">
    Connect to a secure cloud server instance, verifying certificates and decrypting the audio stream with a pre-shared key:
    ```bash
    snapdog-client --server-cert /etc/ssl/certs/ca.pem --encryption-psk "my_secret_key" wss://music.example.com
    ```
  </TabItem>
</Tabs>
